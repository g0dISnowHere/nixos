# CrowdSec On Albaldah

This document describes the repo-managed CrowdSec setup for the `albaldah` VPS.

## Scope

The current setup is owned by the reusable NixOS module at
[modules/nixos/services/crowdsec.nix](../../modules/nixos/services/crowdsec.nix)
and is enabled for `albaldah` from
[flake/machines/servers.nix](../../flake/machines/servers.nix).

It currently covers:

- CrowdSec engine
- local API (LAPI)
- firewall bouncer
- CrowdSec Console enrollment
- CTI API key injection
- automatic CAPI registration when imported online credentials are absent

## Installed Collections

The current module installs these hub collections:

- `crowdsecurity/linux`
- `crowdsecurity/traefik`
- `crowdsecurity/nextcloud`
- `crowdsecurity/http-cve`
- `crowdsecurity/appsec-virtual-patching`
- `crowdsecurity/appsec-generic-rules`
- `crowdsecurity/appsec-crs-exclusion-plugin-nextcloud`
- `firix/authentik`

Intent:

- `linux` keeps the SSH and Linux baseline
- `traefik` covers generic HTTP traffic seen by Traefik
- `nextcloud` adds Nextcloud-specific parsing and scenarios
- `http-cve` adds generic HTTP exploit coverage
- the AppSec collections prepare virtual patching and generic WAF rules
- the Nextcloud AppSec exclusion plugin reduces false positives for Nextcloud
- `firix/authentik` adds Authentik parser and brute-force detection support

Important:

- installing a collection alone does not make it effective
- the related log source or AppSec component must also be wired

## Log Sources

The engine currently acquires SSH logs from `journald` only:

- `_SYSTEMD_UNIT=sshd.service`

This keeps the VPS baseline functional, but only the `linux`/SSH part is fully
active today.

The following still need dedicated acquisitions or component wiring before they
become effective on `albaldah`:

- Traefik access logs
- Nextcloud application logs
- Authentik logs, likely from Docker or a file log
- AppSec request inspection in front of Traefik

## Secret Inputs

The setup expects these repo-managed secret files under
`secrets/services/crowdsec/`:

- `console-enrollment-token.yaml`
  - YAML key: `token`
  - used by `crowdsec-console-enroll.service`
- `cti-api-key.yaml`
  - YAML key: `key`
  - injected into `api.cti.key` through a rendered runtime config

Important:

- `online-api-credentials.yaml` is optional and is only for importing an
  already-existing CrowdSec online credential file
- do not put a CTI API key in `online-api-credentials.yaml`
- on a normal fresh setup, omit `online-api-credentials.yaml` and let
  `crowdsec-capi-register.service` generate
  `/var/lib/crowdsec/online_api_credentials.yaml`

## Runtime Files

The setup manages these important runtime files on the host:

- `/var/lib/crowdsec/local_api_credentials.yaml`
  - local API client credentials
- `/var/lib/crowdsec/online_api_credentials.yaml`
  - generated by `cscli capi register` unless an imported credential file is
    used instead
- `/var/lib/crowdsec/console.yaml`
  - writable Console enrollment/config file
- `/run/secrets/rendered/crowdsec-config.yaml`
  - rendered config used when a CTI key secret exists

## Systemd Units

These units matter during activation and troubleshooting:

- `crowdsec.service`
- `crowdsec-firewall-bouncer.service`
- `crowdsec-firewall-bouncer-register.service`
- `crowdsec-capi-register.service`
- `crowdsec-console-config-init.service`
- `crowdsec-console-enroll.service`
- `crowdsec-update-hub.timer`

Order constraints in the current design:

- `crowdsec-console-config-init.service` prepares `console.yaml` before
  `crowdsec.service`
- `crowdsec-capi-register.service` runs after `crowdsec.service`
- `crowdsec-firewall-bouncer-register.service` waits for
  `crowdsec-capi-register.service` when online credentials are auto-generated

## Deploy

Encrypt or edit the service secrets with `sops`, then deploy:

```bash
sudo nixos-rebuild switch --flake .#albaldah
```

## Verification

Check the key units:

```bash
systemctl status \
  crowdsec \
  crowdsec-firewall-bouncer \
  crowdsec-firewall-bouncer-register \
  crowdsec-capi-register \
  crowdsec-console-config-init \
  crowdsec-console-enroll
```

Inspect logs:

```bash
journalctl -u crowdsec -u crowdsec-firewall-bouncer -u crowdsec-firewall-bouncer-register -u crowdsec-capi-register -u crowdsec-console-config-init -u crowdsec-console-enroll -b
```

Check generated files:

```bash
ls -l /var/lib/crowdsec/local_api_credentials.yaml
ls -l /var/lib/crowdsec/online_api_credentials.yaml
ls -l /var/lib/crowdsec/console.yaml
```

Check CrowdSec state:

```bash
sudo cscli console status
sudo cscli bouncers list
sudo cscli decisions list
sudo cscli metrics
```

## Common Failure Modes

- `crowdsec-console-config-init.service` fails with permission errors
  - the service must be able to create `/var/lib/crowdsec/console.yaml`
- `crowdsec-capi-register.service` fails with permission errors
  - the service must be able to write
    `/var/lib/crowdsec/online_api_credentials.yaml`
- `crowdsec-console-enroll.service` fails with secret read errors
  - the enrollment token secret ownership is wrong for the `crowdsec` user
- `crowdsec-firewall-bouncer-register.service` fails because online credentials
  are missing
  - this is usually downstream of a failed `crowdsec-capi-register.service`
- `crowdsec-firewall-bouncer.service` fails at credential setup
  - usually downstream of failed bouncer registration or missing local API
    bootstrap

## Free Plan Notes

The current repo design assumes the CrowdSec free/community plan still supports:

- Console enrollment by enrollment token
- one or more CTI API keys with community-plan quota
- normal CAPI registration for a security engine

The free plan affects quota and paid features, not the basic wiring documented
here.
